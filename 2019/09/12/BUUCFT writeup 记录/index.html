warmup

进入页面，F12源码，发现提示

访问source.php，得到源码。

从源码中发现了hint.php。

访问该文件，返回 flag not here, and flag in ffffllllaaaagggg。

得到flag，所在位置。

审计得到的源码。

由源码可知，存在PhpMyadmin4.81文件包含的CVE，具体分析可见这篇文章。

构造payload:

http://e94cb492-d91a-463b-98a2-bd1d97fbd65d.node1.buuoj.cn/?file=source.php?../../../../../ffffllllaaaagggg。

得到flag。

原理分析：结合CVE和得到的代码，外面知道，通过在file参数中利用白名单文件加?就可实现绕过，那么最终include得到的参数就是source.php?../../../../../ffffllllaaaagggg，在执行包含文件时，解析器将source.php?看作是一个目录（虽然这个目录是不存在的），所以在后面添加../来跳出目录。【这里还有问题，需要再考虑下，include是如何解析目录的】

总结：

  • PhpMyadmin4.81文件包含的CVE
  • 不安全的使用 in_array() 函数

    in_array ：(PHP 4, PHP 5, PHP 7)

功能 ：检查数组中是否存在某个值
定义 ： bool in_array ( mixed $needle , array $haystack [, bool $strict = FALSE ] )
在 $haystack 中搜索 $needle ，如果第三个参数 $strict 的值为 TRUE ，则 in_array()函数会进行强检查，检查 $needle 的类型是否和 $haystack 中的相同。如果找到 $haystack ，则返回 TRUE，否则返回 FALSE。

PHP在使用 in_array() 函数判断时，会将 7shell.php 强制转换成数字7,如果7在白名单中，就可以实现绕过。
